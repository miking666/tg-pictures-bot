# Полное описание проекта `tg_pictures_bot`

## 1. Назначение проекта

`tg_pictures_bot` — это Telegram-бот для генерации поздравительных открыток по текстовому или голосовому сообщению пользователя.

Проект автоматизирует полный сценарий:

1. получает пожелание от пользователя;
2. при необходимости расшифровывает голосовое сообщение;
3. извлекает короткую надпись для открытки и смысловой контекст;
4. предлагает выбрать визуальный стиль;
5. генерирует изображение через Gemini;
6. проверяет корректность текста на изображении;
7. отправляет готовую открытку в чат;
8. ведет учет примерных расходов на генерации и уведомляет администратора.

Основная ценность проекта — превращение свободного пользовательского ввода в визуально оформленную персонализированную открытку без ручной работы дизайнера.

## 2. Основной пользовательский сценарий

### 2.1. Входные данные

Бот принимает два типа входа:

- обычный текст;
- голосовое сообщение Telegram.

Все остальные типы сообщений считаются неподдерживаемыми.

### 2.2. Сценарий работы

1. Пользователь отправляет текст или голосовое.
2. Если пришло голосовое, бот скачивает файл через Telegram API и отправляет аудио в Gemini на транскрибацию.
3. Полученный текст нормализуется и при необходимости сокращается/уточняется через Gemini:
   - выделяется короткая надпись для открытки;
   - формируется `context_hint` для генерации визуального сюжета.
4. Бот сохраняет промежуточные данные в FSM-состоянии и показывает inline-клавиатуру со стилями.
5. Пользователь выбирает стиль.
6. Бот проверяет бюджет генераций.
7. Бот собирает промпт и отправляет запрос на генерацию изображения в Gemini.
8. Если изображение успешно получено, бот локально накладывает итоговый текст через Pillow.
9. Если внешний генератор недоступен, бот собирает открытку на локальном fallback-фоне и все равно отправляет результат пользователю.
10. После успешной отправки бот обновляет статистику и, если нужно, отправляет предупреждение администратору о бюджете.

## 3. Что проект умеет сейчас

- принимать поздравление в виде текста;
- принимать голосовые сообщения и преобразовывать их в текст;
- выделять краткую надпись для открытки из произвольного пользовательского сообщения;
- учитывать контекст запроса для тематической генерации изображения;
- предлагать выбор из нескольких художественных стилей;
- генерировать тематические фоны через Gemini Image;
- детерминированно рисовать текст локально через Pillow;
- выдавать пользователю рабочую fallback-открытку при сбое внешней генерации;
- вести локальный учет расходов в `.data/budget.json`;
- отправлять администратору статистику, предупреждения и ошибки.

## 4. Границы текущей реализации

Важно понимать текущее фактическое поведение кода.

### 4.1. Как именно появляется текст на открытке

В текущей реализации **Gemini генерирует только фон** и оставляет композиционную зону под надпись. Финальный текст всегда рисуется локально через Pillow, поэтому читаемость и кириллица не зависят от качества рендера модели.

### 4.2. Что происходит при сбое генерации изображения

Локальный резервный рендер фона и наложение текста через Pillow используется как полноценный пользовательский режим деградации. Если внешний генератор не отвечает, бот все равно отправляет открытку на локальном фоне и отдельно уведомляет администратора о причине fallback.

## 5. Архитектура проекта

Проект представляет собой один Python-сервис без отдельной БД и без внешнего очередного процессинга.

### 5.1. Архитектурная схема

- Telegram выступает транспортом пользовательских сообщений.
- `aiogram` обслуживает команды, сообщения и FSM.
- `google-genai` используется для:
  - распознавания голосовых сообщений;
  - извлечения короткого текста открытки и контекста;
  - генерации фонового изображения.
- `Pillow` используется для:
  - приведения изображения к квадратному формату;
  - fallback-отрисовки локального фона;
  - программного наложения текста на финальную открытку.
- Локальные файлы в `.data/` хранят статистику бюджета и FSM-состояние.

### 5.2. Хранилища состояния

В проекте используются два локальных файловых хранилища:

- JSON-storage для пользовательского FSM-состояния;
- `.data/budget.json` для накопленной статистики генераций.

Следствие:

- активные диалоги переживают перезапуск процесса;
- статистика бюджета сохраняется между перезапусками, пока доступна `.data/`.

## 6. Структура репозитория

- `bot.py` — точка входа, роутинг Telegram-событий, FSM, админ-уведомления, orchestration всего сценария.
- `image_gen.py` — сервис генерации открытки, работа с Gemini и локальный рендер текста через Pillow.
- `styles.py` — набор визуальных стилей открыток и их параметры.
- `config.py` — загрузка настроек из `.env` и валидация обязательных переменных.
- `budget.py` — локальный учет расходов, генераций, fallback-статистики и предупреждений.
- `.data/` — runtime-данные проекта, включая бюджет и FSM.
- `README.md` — краткая инструкция по запуску и общая справка.
- `.env.example` — пример конфигурации окружения.
- `requirements.txt` — зависимости Python.
- `fonts/` — шрифт для fallback-отрисовки текста.
- `открытки/` — примеры или референсы изображений.
- `greeting_bot_spec.docx` — исходный документ со спецификацией проекта.

## 7. Детали по модулям

## 7.1. `config.py`

Модуль отвечает за централизованную загрузку настроек.

### Назначение

- читает `.env`;
- проверяет наличие обязательных переменных;
- формирует объект `Settings`;
- нормализует путь к шрифту;
- применяет значения по умолчанию для необязательных настроек.

### Обязательные переменные

- `BOT_TOKEN`
- `GEMINI_API_KEY`

### Необязательные переменные

- `GEMINI_TEXT_MODEL`
- `GEMINI_AUDIO_MODEL`
- `GEMINI_IMAGE_MODEL`
- `CARD_FONT_PATH`
- `CARD_IMAGE_SIZE`
- `MAX_GREETING_LENGTH`
- `MAX_BUDGET_USD`
- `ADMIN_CHAT_ID`

### Дополнительные особенности

- минимальный размер итогового изображения ограничен значением `512`;
- минимальная максимальная длина текста ограничена значением `40`;
- `ADMIN_CHAT_ID` может отсутствовать, тогда админ-уведомления отключаются.

## 7.2. `styles.py`

Модуль содержит словарь `CARD_STYLES` с описанием доступных визуальных режимов.

Каждый стиль задается объектом `CardStyle`:

- `label` — подпись кнопки в Telegram;
- `prompt_fragment` — визуальное описание для Gemini;
- `gradient` — цвета фонового градиента для fallback;
- `accents` — декоративные акцентные цвета для fallback;
- `pattern` — тип декоративного узора в fallback;
- `content_requirement` — дополнительное обязательное требование к содержимому сцены.

### Поддерживаемые стили

| Ключ | Метка | Назначение |
| --- | --- | --- |
| `flowers` | `🌸 Цветы` | флористическая, мягкая, романтичная открытка |
| `portrait` | `👤 Портрет` | изображение с обязательным присутствием человека |
| `animals` | `🐾 Животные` | праздничная открытка с милым животным |
| `fantasy` | `✨ Фэнтези` | сказочная или магическая сцена |
| `festive` | `🎉 Праздничный` | универсальный праздничный сеттинг с подарками и конфетти |
| `landscape` | `🏞️ Пейзаж` | открытка на основе красивого пейзажа |
| `illustration` | `🎨 Иллюстрация` | стилизованная рисованная подача |
| `cozy` | `☕ Уютный` | теплая домашняя атмосфера |

`content_requirement` сейчас особенно важен для режимов `portrait` и `animals`, где модель явно ограничивается по объектному составу сцены.

## 7.3. `budget.py`

Модуль реализует простой файловый контур бюджетного учета.

### Что хранится

В `.data/budget.json` поддерживаются следующие поля:

- `stats_message_id` — id сообщения со статистикой в админ-чате;
- `lifetime` — накопительные счетчики расходов и генераций;
- `events` — события за rolling-window 30 дней для бюджета, fallback и топа пользователей.

### Как считается стоимость

В коде заданы приближенные константы:

- `COST_PER_IMAGE = 0.104`
- `COST_PER_RETRY = 0.104`

Это не строгий биллинг по API-провайдеру, а локальная приблизительная модель расходов.

### Что умеет модуль

- загружать и сохранять бюджет;
- чистить историю старше 30 дней;
- восстанавливать топ пользователей по записям генераций;
- определять, можно ли еще генерировать изображения;
- формировать warning при 80% и 100% исчерпания лимита;
- строить HTML-отчет для админ-чата.

### Ограничения бюджетной модели

- стоимость считается приблизительно;
- неудачные сетевые попытки и часть ошибочных вызовов могут не отражаться в сумме;
- хранение идет в одном локальном JSON-файле без блокировок и без конкурентной защиты.

## 7.4. `image_gen.py`

Это центральный прикладной модуль проекта.

### Основные обязанности

- создать Gemini-клиент;
- транскрибировать аудио;
- извлечь итоговую короткую фразу и контекст из пользовательского текста;
- сгенерировать изображение;
- повторить генерацию при проблемах с текстом;
- при необходимости построить локальный fallback-фон;
- локально наложить текст на изображение в fallback-ветке.

### Ключевые сущности

- `TextLayout` — параметры раскладки текста для Pillow.
- `RenderedCard` — результат генерации:
  - `image_bytes`
  - `used_fallback_background`
  - `fallback_reason`
- `BudgetExceededError` — исключение при исчерпанном бюджете.
- `GreetingCardService` — основной сервис генерации.

### Внутренний пайплайн `GreetingCardService`

#### `transcribe_audio(...)`

Отправляет байты голосового сообщения в Gemini и ожидает только распознанный текст.

#### `refine_greeting(...)`

Использует текстовую модель Gemini, чтобы из произвольного сообщения получить:

- краткую надпись для открытки;
- подробный англоязычный контекст для визуальной генерации.

Код заставляет модель отвечать строго в формате:

- `GREETING: ...`
- `CONTEXT: ...`

Это нужно, чтобы отделить короткую фразу на картинке от визуального описания сцены.

#### `create_card(...)`

Выполняет общий сценарий сборки открытки:

1. проверяет бюджет;
2. запускает генерацию фона;
3. при успехе приводит фон к квадратному кадру;
4. локально рисует итоговый текст поверх изображения;
5. при сбое внешней генерации переключается на локальный fallback;
6. возвращает флаг деградации и причину fallback.

#### `_build_background(...)`

Делает до 4 попыток сгенерировать изображение через Gemini. При успехе фиксирует бюджет и возвращает изображение. Если все попытки провалились, строит локальный fallback и помечает результат как деградировавший.

#### `_generate_image_gemini(...)`

Собирает итоговый промпт из:

- базового quality preset;
- композиционного seed;
- атмосферных edge elements;
- требований стиля;
- контекстной подсказки;
- инструкции по резервированию свободной зоны под локальную надпись.

Промпт специально сформулирован так, чтобы модель не рисовала никакой текст и оставляла визуально спокойную область для последующего локального оверлея.

#### Локальный fallback-рендер

В модуле реализованы:

- градиентный фон;
- декоративные узоры (`confetti`, `petals`, `snow`, `watercolor`, `sunset`);
- динамический подбор размера шрифта;
- перенос текста по строкам;
- скругленная подложка под текст;
- размытие декоративных и фоновых слоев.

Fallback используется как штатный пользовательский режим деградации и отправляется в чат как рабочий финальный результат.

## 7.5. `bot.py`

Модуль управляет жизненным циклом Telegram-бота.

### Основные обязанности

- инициализация `Settings` и `GreetingCardService`;
- настройка `aiogram`-роутера и FSM;
- прием сообщений и callback-запросов;
- обработка ошибок;
- взаимодействие с бюджетным контуром;
- отправка уведомлений администратору.

### Состояния

Используется `GreetingFlow.waiting_for_style`.

Это означает, что после получения текста бот ждет ровно один следующий логический шаг: выбор стиля.

### Обработчики

- `/start` — очищает состояние и объясняет базовый сценарий.
- `/help` — краткая инструкция по использованию.
- `/cancel` — сброс текущего сценария.
- `F.voice` — загрузка и распознавание голосового.
- `F.text` — обработка текстового поздравления.
- `callback style:*` в нужном состоянии — запуск генерации открытки.
- `callback style:*` вне состояния — сообщение о том, что сначала нужен текст.
- fallback message handler — ответ для неподдерживаемых типов сообщений.

### Административная логика

В коде есть middleware `StatsAccountFilter`, которое блокирует обычное использование бота с аккаунта, указанного в `ADMIN_CHAT_ID`.

Этот же admin id используется для:

- получения текущей статистики;
- получения предупреждений по бюджету;
- получения stack trace и текста ошибок.

### Отказоустойчивость

- ошибки транскрибации голосового не роняют процесс;
- ошибки генерации логируются и отправляются администратору;
- отправка изображения в Telegram ретраится до 3 раз при `TelegramNetworkError`;
- при проблемах с редактированием/удалением сообщений часть ошибок подавляется, чтобы не ломать основной сценарий.

## 10. Формат данных и состояние

### FSM-состояние пользователя

В FSM сохраняются:

- `context_hint` — дополнительный визуальный контекст для генерации.

Эти данные сохраняются в локальном JSON-storage и переживают перезапуск процесса.

### Структура `.data/budget.json`

Практически файл хранит:

- общие счетчики;
- id сервисного сообщения со статистикой;
- агрегированный список пользователей;
- журнал отдельных генераций с временными метками.

Файл одновременно выполняет роль:

- локальной БД;
- простого аудита активности;
- источника данных для админ-дашборда в Telegram.

## 11. Работа с ошибками и деградация

В проекте предусмотрены несколько уровней обработки ошибок.

### Пользовательский уровень

- при пустом сообщении бот просит прислать текст;
- при слишком длинном сообщении просит сократить текст;
- при проблеме с голосовым предлагает повторить попытку или прислать текст;
- при неизвестном стиле показывает alert;
- при потере FSM-данных просит прислать текст заново.

### Системный уровень

- исключения логируются через `logging`;
- администратору отправляется короткий traceback;
- бюджетные предупреждения формируются автоматически;
- при проблеме генерации кнопки выбора стиля восстанавливаются.

### Важный нюанс деградации

Если Gemini недоступен, перегружен или не вернул картинку:

- локально строится fallback-фон;
- причина сбоя упаковывается в `fallback_reason`;
- бот не завершает сценарий успешно;
- пользователю предлагается попробовать заново.

Для production это означает, что текущая отказоустойчивость ориентирована скорее на сохранение стабильности процесса, чем на гарантированную выдачу картинки при сбое внешнего провайдера.

## 12. Наблюдаемость и администрирование

Если указан `ADMIN_CHAT_ID`, проект поддерживает три канала админ-обратной связи:

1. статистика расходов и генераций;
2. предупреждения о приближении к лимиту бюджета;
3. сообщения об ошибках с данными пользователя и traceback.

Статистика отправляется и затем обновляется через редактирование уже существующего сообщения, id которого хранится в `.data/budget.json`.

## 13. Приватность и данные

Проект обрабатывает пользовательские данные, о которых важно помнить:

- текст сообщений пользователей отправляется в Gemini;
- голосовые сообщения пользователей отправляются в Gemini;
- в `.data/budget.json` сохраняются id пользователей, имена и время генераций;
- при ошибках администратору может отправляться информация о пользователе, вызвавшем сбой.

Для реального использования это означает необходимость отдельно определить политику хранения и обработки персональных данных.

## 14. Технические ограничения текущей версии

- тестовое покрытие есть только для критичных локальных сценариев и пока не закрывает весь bot-flow end-to-end;
- нет очередей задач и фоновых воркеров;
- нет rate limiting по пользователям;
- нет отдельной панели администратора;
- нет строгого учета фактической стоимости API-вызовов;
- проект рассчитан на запуск как один процесс с polling;
- код не упакован как сервис с контейнеризацией, systemd-конфигом или CI/CD.

## 15. Что уже сделано хорошо

- код разделен по ролям на небольшие модули;
- конфигурация вынесена в отдельный dataclass;
- пользовательский сценарий простой и линейный;
- есть базовая защита бюджета;
- есть повторные попытки и восстановление после части сбоев;
- реализованы уведомления администратору;
- есть продуманные визуальные стили и достаточно сильный prompt engineering слой.

## 16. Наиболее очевидные направления развития

Если проект развивать дальше, самые логичные шаги такие:

1. перевести FSM и бюджетное состояние из локальных JSON-файлов во внешнее хранилище;
2. добавить тесты на конфигурацию, бюджет и ключевые ветки `GreetingCardService`;
3. вынести стоимость запросов в настраиваемую модель или реальный биллинг;
4. добавить ограничение по частоте запросов и защиту от злоупотреблений;
5. улучшить эксплуатационный контур: structured logging, health checks, контейнеризация;
6. при необходимости перейти с polling на webhook-интеграцию.

## 17. Краткий итог

`tg_pictures_bot` — это прикладной Telegram-бот для генерации персонализированных поздравительных открыток, построенный вокруг `aiogram`, Gemini и Pillow.

С точки зрения продукта он уже закрывает основной пользовательский сценарий end-to-end: от входного текста или голосового сообщения до итоговой открытки и административного контроля расходов.

С точки зрения эксплуатации это пока компактный single-process проект с локальными файлами состояния, хорошим базовым UX и заметным потенциалом для усиления надежности, персистентности и production-ready инфраструктуры.
